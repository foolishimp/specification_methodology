# Release v1.0.1

**Repository**: `specification_methodology`
**Tag**: `v1.0.1`
**Release Commit**: `009b5de`
**Release Date**: 2026-04-22

## Position

`v1.0.1` is the first post-`v1.0.0` method tightening release.

This cut strengthens migration law for interface-changing work and adds an
explicit realization-method surface for low-coupling, prime-compressed module
design.

## Included In This Release

- `SPEC_METHOD.md` now distinguishes two constitutional migration strategies:
  `inside_out_hard_break` and `fundamental_re_adoption`
- inside-out migration is now defined as a hard-break sequence over one
  interface family, with explicit seam severing, keep-broken discipline, and
  per-break negative proof
- fundamental migration now requires explicit adjudication of inherited
  implementation as `carry_across`, `redundant`, or `rewrite`
- `TICKET_METHOD.md` now requires `migration_strategy` for
  `implementation_migration` tickets and adds stronger checklist law for
  impacted interfaces, break order, and negative proof
- `DESIGN_MODULE_METHOD.md` is added as an approved realization-method surface
  covering low coupling, immutable carriers, explicit effect boundaries, and
  the Prime Law
- `specification/standards/README.md` is updated so the standards library lists
  `DESIGN_MODULE_METHOD.md` as part of the shared methodology set

## Key Outcomes

This release materially tightens the method in four places:

1. migration sequencing is now explicitly break-first rather than advisory
2. major rewrites can now be governed as sideways re-adoption instead of
   in-place partial carry-forward
3. ticket closure claims must now name migration strategy, impacted interfaces,
   and real negative proof rather than relying on green tests alone
4. projects that adopt the design method now have a named standard for prime
   compression, low coupling, and effect-edge realization
