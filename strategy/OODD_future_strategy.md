# OODD Future Strategy

**Status**: Supplemental future strategy
**Authority**: Supporting documentation only; not constitutional authority
**Scope**: Organisational operating model built on Outcome Driven Development
**Relation**: Extends the current `odd_sdlc` outcome-driven development line

---

## Position

`ODD` defines legitimate work against outcome states.

`OODD` is the organisational form that operates on that method.

The distinction is useful:

- `ODD` explains how work becomes legitimate
- `OODD` explains how an organisation is structured around that work model

This note treats `OODD` as a supporting future-strategy surface for
`odd_sdlc`.

It is not constitutional method yet.

It is a working model for what a graph-native electronic organisation could be.

---

## Core Claim

`OODD` is an electronic organisation built from recursive outcome-governed
cells.

Each cell:

- has its own bounded context
- owns its own local outcome surfaces
- receives work through declared interfaces
- produces closure through evidence
- may create child cells beneath the same rules

That makes `OODD`:

- hierarchical in governance
- recursive in method
- graph-shaped in dependency

All three properties matter.

Hierarchy gives authority.

Recursion gives scale.

Graph structure gives the real dependency map.

---

## Why OODD Exists

Human organisations usually fuse several things together:

- governance
- custody
- execution
- staffing
- memory
- reporting

That makes them expensive to maintain and hard to reason about.

`OODD` separates those concerns.

It keeps:

- governance durable
- boundaries strict
- execution elastic
- proof explicit

The organisation persists as declared outcome boundaries, work contracts, and
evidence surfaces.

Execution can be created on demand inside those boundaries.

That is the efficiency advantage.

The boundary stays stable.

The work cell inside it does not need to.

## OODD Stack

The current conceptual stack is:

- `SPEC_METHOD` — philosophical baseline
- `ODD_METHOD` — constitutional graph-native method and ODD product-authoring law (absorbed the prior `GRAPH_METHOD` on 2026-04-21)
- `ODD` — work method defined against outcomes, realised by `ODD_METHOD` + product-local specification
- `OODD` — organisational operating model built on that work method
- `GTL` — language expression of graph-native method
- `ABG` — execution and binding runtime

`OODD` sits above the language and runtime.

It is the organisational pattern those lower layers make possible.

## Cell Model

The unit of organisation in `OODD` is the cell.

A cell is a bounded organisational context with:

- a declared purpose
- a declared outcome surface
- declared interfaces to parent, peers, and children
- local work vectors and graph functions
- local evidence and proving surfaces

Each cell may be small or large.

It may correspond to:

- one session
- one work wave
- one feature
- one subsystem
- one product
- one release program
- one whole organisation

The scale changes.

The method does not.

---

## Recursive Pattern

Each cell can run the same pattern internally:

1. define goals
2. derive intent
3. define the product or outcome surface
4. decompose obligations
5. choose work vectors
6. execute graph functions
7. prove closure

That means the organisation is recursive.

The same operating model can hold at every scale.

The organisation is therefore not a flat workflow with many tasks.

It is a hierarchy of cells that reuse the same method.

---

## Hierarchy

The hierarchy in `OODD` is not there for reporting lines.

It is there for authority.

A parent cell should govern a child through:

- declared outcome contracts
- accepted evidence surfaces
- escalation rules
- custody boundaries

The parent does not need the child's full internal noise.

It needs the boundary contract and the proof that the contract closed.

That keeps governance bounded.

It also prevents the parent from collapsing the child into raw micromanaged
activity.

---

## Graph

The organisation is not a tree in dependency terms.

It is a graph.

Cells may depend on shared assets, shared proving surfaces, and multi-source
transitions.

That is why `OODD` needs both hierarchy and graph structure:

- hierarchy governs authority
- graph structure governs work dependencies

The hierarchy alone is not enough because real work converges and reuses.

The graph alone is not enough because authority and custody need boundaries.

`OODD` needs both at once.

---

## Markov Blankets And Bounded Contexts

The strongest way to think about a cell is as a bounded context or Markov
blanket.

Each cell has:

- internal state
- an exposed interface
- controlled signals across the boundary
- local coherence inside the boundary

That gives a precise organisational model.

Inside the blanket:

- drafts
- revisions
- discussion
- local iteration
- local decomposition

Across the blanket:

- commitments
- requests
- supplied assets
- evidence
- escalation

This is a stronger boundary than an informal team boundary.

It makes the organisation legible and enforceable.

---

## Strict Boundaries, Elastic Execution

This is one of the main organisational advantages of `OODD`.

The boundary can stay strict while the execution inside it stays elastic.

That means:

- the cell contract persists
- the custody boundary persists
- the evidence boundary persists
- the work inside can be spun up on demand
- the work cell can collapse when the outcome closes

Human organisations usually keep both the boundary and the workforce
persistent.

`OODD` does not need to.

It can preserve the contract and recreate the execution body when needed.

This lets the organisation be more efficient without weakening control.

The stable thing is not the staff shape.

The stable thing is the outcome boundary.

---

## On-Demand Applications And Dedicated Releases

One useful consequence of `OODD` is that an application can be operated as an
on-demand bounded delivery path.

A request enters as a candidate outcome contract.

The organisation can then:

- gate the request
- determine whether it targets a new application or an existing one
- bind the lawful input assets and constraints
- spin up the needed execution cell
- run the SDLC to release closure
- deploy the resulting release

The same pattern works for existing applications when the requester wants a
version tuned to local preference.

In that case the organisation can derive a dedicated release for one operator,
customer, team, or environment while preserving provenance to the shared base
line.

That makes tuning a governed release activity rather than an informal fork.

The requester receives a release line shaped to local taste, workflow, policy,
or interface preference.

The governing product line keeps boundary clarity and can choose what, if
anything, is later adopted back.

---

## Delegation

Delegation in `OODD` should be contractual.

The parent cell delegates by naming:

- the required outcome state
- the lawful input assets
- the boundary conditions
- the evidence required for closure

The child cell then resolves the internal work using its own local method.

That is a cleaner delegation model than assigning tasks without outcome
contracts.

It also means a child cell can be human, agentic, mixed, persistent, or
temporary.

The parent only needs the contract to hold.

---

## Organisational Memory

`OODD` needs organisational memory, but that memory must not corrupt live
truth.

That means the organisation should preserve distinctions between:

- live authority
- superseded authority
- historical discussion
- runtime evidence
- release cuts
- local internal state

This is not a documentation concern.

It is an organisational integrity concern.

If historical discussion leaks into live authority, the organisation loses its
boundary discipline.

The same is true if runtime state silently becomes product law.

---

## OODD And Self-Modification

`OODD` makes self-modification easier to reason about because it gives the
builder and the built product separate organisational blankets.

For lawful self-modification, the organisation must preserve distinct cells
for:

- the released builder in use
- the in-development product being built
- the proving surface that checks the boundary held

If those collapse into one mixed blanket, authority fails.

That means self-modification in `OODD` is not "the system edits itself."

It is:

- one bounded cell building another bounded cell
- under declared contracts
- with explicit proving of the separation

That is the compiler analogy in organisational form.

One released compiler builds the next compiler release.

The two do not become the same thing mid-build.

---

## Recursive LLMs

The connection to recursive LLM systems is direct.

A recursive LLM system is not only one loop calling itself.

It is a stack of bounded contexts that can decompose work into smaller bounded
contexts using the same method.

`OODD` is the organisational reflection of that pattern.

The cell is the organisational equivalent of a recursive bounded context.

That means:

- parent cells define outer contracts
- child cells solve bounded local problems
- proofs flow back up the hierarchy
- method stays stable across scales

This is how a large electronic organisation can stay coherent while remaining
highly elastic.

---

## Work Vectors In OODD

`ODD` introduced the idea of the work vector.

`OODD` gives that idea an organisational place.

A work vector is a productized work capability selected because of its
characteristics and guarantees.

In organisational terms, that means the organisation can maintain a registry
of work vectors such as:

- consensus and revision closure over a document
- task-list generation with traceability
- qualification and proving bundles
- decomposition of outcome surfaces into lower-level obligations
- bounded repair loops over failing surfaces

The cell does not invent procedure from scratch every time.

It chooses from known work vectors that fit the graph and the required outcome.

That is how the organisation accumulates useful building blocks without losing
legitimacy.

---

## OODD And Efficiency

The efficiency claim in `OODD` is specific.

It is not "automation makes things faster."

It is:

- durable governance boundaries
- durable proof surfaces
- reusable work vectors
- elastic execution inside bounded cells

That combination is what lets an electronic organisation outperform a human
organisation on many classes of work.

The organisation does not pay the full cost of permanent internal structure
just to keep order.

Order is carried by the declared outcome system.

Execution is attached when needed.

---

## What OODD Rejects

`OODD` rejects several organisational habits:

- permanent structure without active outcome contracts
- delegation without declared closure criteria
- authority surfaces mixed with runtime noise
- self-modification without boundary proof
- organisational memory that rewrites live truth
- task assignment as a substitute for product outcome definition

These habits produce drift.

`OODD` only works if the boundaries stay legible.

---

## Immediate Implications

If `OODD` is the right direction, several things become important:

- stronger custody and provenance at cell boundaries
- better declaration of work vectors and their guarantees
- explicit parent-child outcome contracts
- cleaner distinction between live authority and internal working state
- explicit proving surfaces for self-host separation
- stronger runtime support for replayable closure evidence

This pushes on `GTL`, `ABG`, and the concrete domain applications built on them.

The method pressure is not accidental.

The organisational model is asking the language and runtime to become equal to
the structure they are meant to carry.

---

## Working Definition

`Organisation Outcome Driven Development` is a recursive, hierarchical,
graph-native electronic organisation built from bounded outcome-governed cells.

Each cell preserves strict boundaries, executes work through lawful work
vectors and graph functions, and proves closure through evidence.

In short:

- `ODD` defines legitimate work
- `OODD` defines the organisation that runs on that work model
- hierarchy governs authority
- graph structure governs dependencies
- recursion lets the same method operate at every scale
- strict boundaries and elastic execution make the organisation efficient

That is the working strategy for `OODD`.
